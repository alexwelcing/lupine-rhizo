# LAMMPS log → versioned evidence → Lean theorems

You run LAMMPS on your own cluster; this directory shows how the log files you
already produce become versioned `lupine.mlip.lammps_evidence.v1` payloads and
machine-checked Lean 4 theorems. No knowledge of our stack required — the entry
point is a plain CLI over files.

Setup (once, from the repo root):

```bash
pip install -e python
```

## Run the demo on the bundled sample logs

```bash
python3 hpc/examples/lammps_to_lean_demo.py
```

## Run against your own log files

Parse your `examples/ELASTIC`-driver log (supply the reference values you want
the claims checked against, with a citation):

```bash
python3 -m lupine_distill.lammps_ingest parse /path/to/your/log.lammps \
    --material Ni --potential Ni_u3.eam --input-script in.elastic \
    --ref C11=246.5 --ref C12=147.3 --ref C44=124.7 \
    --ref-source "Simmons & Wang 1971" \
    -o my_evidence.json
```

A plain thermo log (MD run) is summarized with `--kind thermo`; a separate
thermo log can be attached to an elastic payload with `--thermo-log PATH`.
Then emit the Lean module (default tolerance: 5% of |reference|):

```bash
python3 -m lupine_distill.lammps_ingest lean my_evidence.json -o generated/MyModule.lean
```

`--help` on either subcommand documents all flags. A directory of evidence
payloads also plugs straight into the benchmark machinery:

```python
from lupine_distill.runner import build_backend
backend = build_backend("lammps", model_id="Ni_u3.eam", evidence_dir="path/to/payloads")
```

## What each artifact is

- `sample_logs/ni_eam_elastic.log` — synthetic (clearly marked) `examples/ELASTIC`
  driver output for fcc Ni / `Ni_u3.eam`, C11/C12/C44 near 247/147/125 GPa.
- `sample_logs/ni_eam_thermo.log` — synthetic short NVT thermo log, same system.
- `generated/*.evidence.json` — schema-validated `lupine.mlip.lammps_evidence.v1`
  payloads: parsed properties + your references, thermo summary, and provenance
  (sha256 of the log text; a timestamp only if you pass `--parsed-at`).
- `generated/Ni_EAM.lean` — one decidable Nat-inequality theorem per
  reference-annotated property (abs error vs reference, x1000-scaled), verdict
  encoded either way (`within_tol` / `exceeds_tol`), zero `sorry`.

## How a generated module gets admitted into lean-spec

Generated modules land in `hpc/examples/generated/` and are NOT part of the
Lean build. Admission is a reviewed step: the module is regenerated by the
production tool (`tools/mlip_distill_atlas.py` is the existing path that writes
`lean-spec/OpenDistillationFactory/Materials/DistillAtlas/`), placed under
`lean-spec/`, imported by the root aggregator (`Vision.lean`), and then `lake
build` must pass the zero-sorry gate — every theorem is `by decide`, so the
checker either accepts the claim or rejects the module. Only after that build
passes does the evidence count as machine-checked.
